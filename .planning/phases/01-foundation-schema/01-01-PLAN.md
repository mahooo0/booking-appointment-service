## Existing Patterns

From prisma/schema.prisma (existing models):
```prisma
model Appointment {
  id                   String            @id @default(uuid())
  organizationId       String
  branchId             String
  // ... other fields
  @@index([organizationId])
  @@map("appointments")
}

model ServiceDuration {
  id              String   @id @default(uuid())
  organizationId  String
  branchId        String?
  // ... other fields
  @@index([organizationId])
  @@map("service_durations")
}
```

**Key Pattern:** Codebase uses `organizationId` (NOT `companyId`) for tenant isolation.

From src/common/decorators/account-id.decorator.ts:
```typescript
export const AccountId = createParamDecorator(
  (data: unknown, ctx: ExecutionContext): string => {
    const request = ctx.switchToHttp().getRequest();
    const accountId = request.headers['x-account-id'] as string;
    // Returns organizationId value
    return accountId;
  },
);
```

## Tenant Field Naming Decision

**Decision (made during planning):** Use `organizationId` to match existing Appointment/ServiceDuration models. This avoids breaking changes to existing schema and aligns with AccountId decorator extracting from x-account-id header.

**Rationale:**
- Existing models (Appointment, ServiceDuration) already use `organizationId`
- AccountId decorator extracts organizationId from x-account-id header
- Changing to `companyId` would require migrating existing tables (breaking change)
- REQUIREMENTS.md uses `companyId` terminology, but this is a documentation correction, not a functional change

**Impact:** All new models in this phase will use `organizationId` field for tenant isolation.

**Schedule Model (SCHEMA-02):**
- id (String, UUID primary key)
- organizationId (String)
- specialistId (String)
- locationId (String)
- dayOfWeek (Int) - 0=Sunday, 6=Saturday
- intervals (Json) - Array of {startTime: "HH:MM", endTime: "HH:MM"}
- isDayOff (Boolean, default false)
- timestamps (createdAt, updatedAt)
- Foreign keys:
  - specialist Specialist @relation(fields: [specialistId], references: [id], onDelete: Cascade)
  - location Location @relation(fields: [locationId], references: [id], onDelete: Cascade)
- Unique constraint: @@unique([organizationId, specialistId, locationId, dayOfWeek])
- Indexes: [organizationId, specialistId, locationId], [organizationId, locationId, dayOfWeek]
- Map to "schedules" table

**Why onDelete: Cascade:** When specialist or location is deleted, their schedules should auto-delete (no orphaned schedules). Research confirms this is safe and prevents manual cleanup logic.

**Why Json for intervals:** Research (01-RESEARCH.md Pattern 4) recommends JSON array for 1-5 intervals per day. Simpler than separate ScheduleInterval table, atomic updates, covers 95% of use cases.
