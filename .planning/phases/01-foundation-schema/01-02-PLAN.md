## Research Pattern: Explicit Junction Tables

From 01-RESEARCH.md Pattern 1:
```prisma
model SpecialistService {
  id           String   @id @default(uuid())
  companyId    String   // Multi-tenant isolation
  specialistId String
  serviceId    String
  createdAt    DateTime @default(now())

  specialist Specialist @relation(fields: [specialistId], references: [id], onDelete: Cascade)
  service    Service    @relation(fields: [serviceId], references: [id], onDelete: Cascade)

  @@unique([companyId, specialistId, serviceId])
  @@index([companyId, specialistId, serviceId])

  @@map("specialist_services")
}
```

**Why explicit (not implicit):**
- Supports tenant field (companyId/organizationId) for multi-tenant isolation
- Allows relationship metadata (createdAt, assignedBy, etc.)
- Prevents implicit-to-explicit migration data loss (Critical Pitfall #2 from research)

## Research Pattern: Composite Index Ordering

From 01-RESEARCH.md Pattern 2:
- **Always put tenant field first** in composite indexes
- Add bidirectional indexes for reverse lookups (e.g., specialist→services AND service→specialists)
- **150× performance improvement** vs single-column indexes for tenant-scoped queries

**Example:**
```prisma
@@index([organizationId, specialistId, serviceId]) // Forward: specialist's services
@@index([organizationId, serviceId, specialistId]) // Reverse: service's specialists
```

<task type="auto">
  <name>Task 2: Add Composite Indexes and Unique Constraints</name>
  <files>prisma/schema.prisma</files>
  <action>
Add composite indexes (SCHEMA-06) and unique constraints (SCHEMA-07) to all three junction tables per research recommendations.

**For SpecialistService table:**
- Unique constraint: `@@unique([tenantField, specialistId, serviceId])` - prevents duplicate specialist-service relationships within tenant
- Forward index: `@@index([tenantField, specialistId, serviceId])` - "get all services for specialist X in tenant Y" query
- Reverse index: `@@index([tenantField, serviceId, specialistId])` - "get all specialists for service X in tenant Y" query
- Tenant-only index: `@@index([tenantField, specialistId])` - partial lookups when serviceId not specified

**For SpecialistLocation table:**
- Unique constraint: `@@unique([tenantField, specialistId, locationId])` - prevents duplicate specialist-location relationships
- Forward index: `@@index([tenantField, specialistId, locationId])` - "get all locations for specialist X"
- Reverse index: `@@index([tenantField, locationId, specialistId])` - "get all specialists at location X"
- Tenant-only index: `@@index([tenantField, specialistId])` - partial lookups

**For ServiceLocation table:**
- Unique constraint: `@@unique([tenantField, serviceId, locationId])` - prevents duplicate service-location relationships
- Forward index: `@@index([tenantField, serviceId, locationId])` - "get all locations offering service X"
- Reverse index: `@@index([tenantField, locationId, serviceId])` - "get all services at location X"
- Tenant-only index: `@@index([tenantField, serviceId])` - partial lookups

**Why tenant field first in all indexes:**
Multi-tenant apps ALWAYS filter by tenant first. PostgreSQL can use index prefix even if not all columns match query (Research Pattern 2). Placing tenant field first ensures fast tenant isolation without full table scans.

**Why bidirectional indexes:**
Supports both "specialist's services" and "service's specialists" queries without forcing full table scans on reverse direction.

**Why unique constraints separate from indexes:**
Unique constraint enforces data integrity at database level (handles concurrent INSERT race conditions). Index optimizes query performance. They serve different purposes (Research SCHEMA-07).
